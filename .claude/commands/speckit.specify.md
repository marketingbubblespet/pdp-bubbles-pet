# Spec-Kit: Specify

Generate a structured specification for: **$ARGUMENTS**

## Pre-Flight

### Git status
!`$HOME/.claude/hooks/speckit-helper.sh branch`

### Constitution
!`$HOME/.claude/hooks/speckit-helper.sh constitution`

### Existing specs
!`$HOME/.claude/hooks/speckit-helper.sh list-specs-dir`

### Project structure
!`$HOME/.claude/hooks/speckit-helper.sh detect-stack`

## Instructions

1. **Verify `.specify/` exists** — if not, tell the user to run `/speckit.init` first

2. **Generate branch name**
   - Format: `short-description` (2-4 words, kebab-case, no sequential numbering)
   - Must be unique among existing `specs/` directories
   - Example: `chat-system`, `user-auth`, `fxblue-auto-download`

3. **Create spec directory**: `.specify/specs/<branch-name>/`

4. **Write `spec.md`** using the `spec-template` skill patterns:

   ```markdown
   # Spec: <branch-name>
   <!-- Generated by /speckit.specify -->
   <!-- Date: YYYY-MM-DD -->

   ## Overview
   <1-2 sentence description of the feature>

   ## User Scenarios

   ### US1: <Scenario Name> [P1]
   - **Given** <precondition>
   - **When** <action>
   - **Then** <expected outcome>

   ### US2: <Scenario Name> [P2]
   - **Given** <precondition>
   - **When** <action>
   - **Then** <expected outcome>

   <!-- Add US3+ as needed. Prioritize: P1=must, P2=should, P3=nice-to-have -->

   ## Functional Requirements

   | ID | Requirement | Priority | Scenario |
   |----|-------------|----------|----------|
   | FR-001 | <requirement> | P1 | US1 |
   | FR-002 | <requirement> | P1 | US1 |
   | FR-003 | <requirement> | P2 | US2 |

   ## Success Criteria

   | ID | Criterion | Validation Method |
   |----|-----------|-------------------|
   | SC-001 | <measurable criterion> | <how to verify> |
   | SC-002 | <measurable criterion> | <how to verify> |

   ## Clarifications
   <!-- Max 3 [NEEDS CLARIFICATION] markers allowed -->
   _None yet._
   ```

5. **Limit `[NEEDS CLARIFICATION]` markers** to a maximum of 3. For anything beyond that, make a reasonable decision and document it in the spec.

6. **Auto-generate `checklists/requirements.md`** in the spec branch directory:
   ```markdown
   # Requirements Checklist: <branch-name>
   <!-- Auto-generated from spec.md by /speckit.specify -->

   | ID | Requirement (from FR) | Quality Check | Status |
   |----|----------------------|---------------|--------|
   | CHK001 | FR-001: <brief> | [completeness] | [ ] |
   | CHK002 | FR-001: <brief> | [testability] | [ ] |
   ```

7. **Create git branch** (if in a git repo):
   - Branch name: `feature/<branch-name>` (e.g., `feature/003-chat-system`)
   - Create from current HEAD
   - Do NOT switch to it yet — ask user first

8. **Present summary** to user:
   - Number of user scenarios, functional requirements, success criteria
   - Any `[NEEDS CLARIFICATION]` items
   - Proposed branch name
   - Suggest next step: `/speckit.plan` or `/speckit.clarify` if there are open questions
