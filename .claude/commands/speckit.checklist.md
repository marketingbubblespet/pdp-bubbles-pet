# Spec-Kit: Checklist

Generate requirement quality checklists (NOT implementation tests) for the current branch's specification.

## Pre-Flight

### Current branch
!`$HOME/.claude/hooks/speckit-helper.sh branch`

### Load spec
!`$HOME/.claude/hooks/speckit-helper.sh spec`

### Existing checklists
!`$HOME/.claude/hooks/speckit-helper.sh checklists-dir`

## Instructions

1. **Load spec.md** and analyze functional requirements and user scenarios

2. **Generate checklists** by type. Create each as a separate file in `.specify/specs/<branch>/checklists/`:

   ### `requirements.md` — Requirement Quality
   Validates that requirements are well-formed before implementation begins.

   | Dimension | What it checks |
   |-----------|---------------|
   | [completeness] | All necessary details present |
   | [consistency] | No contradictions between requirements |
   | [testability] | Can be verified with a concrete test |
   | [clarity] | Unambiguous, specific language |
   | [feasibility] | Technically achievable within constraints |

   ### `ux.md` — User Experience (if feature has UI)
   | Dimension | What it checks |
   |-----------|---------------|
   | [accessibility] | WCAG compliance considerations |
   | [responsiveness] | Mobile/desktop behavior defined |
   | [feedback] | User gets clear feedback for actions |
   | [error-states] | Error messages and recovery flows |

   ### `api.md` — API Quality (if feature has API endpoints)
   | Dimension | What it checks |
   |-----------|---------------|
   | [contract] | Request/response shapes defined |
   | [validation] | Input validation rules specified |
   | [errors] | Error response formats defined |
   | [versioning] | Breaking change policy addressed |

   ### `security.md` — Security (if feature handles auth/data)
   | Dimension | What it checks |
   |-----------|---------------|
   | [authentication] | Auth requirements defined |
   | [authorization] | Permission model specified |
   | [data-protection] | Sensitive data handling addressed |
   | [input-sanitization] | Injection prevention considered |

3. **Checklist format** for each file:

   ```markdown
   # Checklist: <type>
   <!-- Generated by /speckit.checklist -->
   <!-- Branch: <branch-name> | Date: YYYY-MM-DD -->

   | ID | Check | Dimension | FR/US Ref | Status |
   |----|-------|-----------|-----------|--------|
   | CHK001 | <specific check description> | [completeness] | FR-001 | [ ] |
   | CHK002 | <specific check description> | [testability] | FR-002 | [ ] |
   | CHK003 | <specific check description> | [clarity] | US1 | [ ] |
   ```

4. **Rules**:
   - Only generate checklist types relevant to the feature (skip `ux.md` for backend-only features, skip `api.md` for UI-only features)
   - `requirements.md` is ALWAYS generated
   - CHK IDs are globally unique across all checklist files (CHK001-CHK999)
   - Every check must reference at least one FR-NNN or US#
   - Checks should be specific enough that pass/fail is unambiguous

5. **Present summary** to user:
   - Checklist types generated
   - Total checks per type
   - Suggest: review and check off items manually, or run `/speckit.implement` which validates these as a pre-implementation gate
