# Spec-Kit: Baseline

Reverse-engineer a specification from existing code in: **$ARGUMENTS**

## Pre-Flight

### Current branch
!`$HOME/.claude/hooks/speckit-helper.sh branch`

### .specify directory
!`$HOME/.claude/hooks/speckit-helper.sh check-specify-dir`

### Existing code
!`$HOME/.claude/hooks/speckit-helper.sh detect-existing-code`

### Constitution
!`$HOME/.claude/hooks/speckit-helper.sh constitution`

## Instructions

**Required**: `.specify/` must exist. If not, tell the user to run `/speckit.init` first.

This command analyzes existing code to generate a spec.md, enabling the SDD workflow for brownfield projects or existing modules that lack formal specifications.

### Step 1: Inventory

- Read source files in the target module/directory (or whole project if no argument)
- Identify: public interfaces, exported functions, API endpoints, data models
- Map the module structure: which files do what, how they relate
- Note the tech stack, frameworks, and patterns in use

### Step 2: Extract Behaviors

From the code, infer:
- **User scenarios** (Given/When/Then): What does this code enable a user to do?
- **Functional requirements**: What business rules does the code enforce?
- **Success criteria**: What measurable outcomes does the code produce?
- **Error handling**: What error conditions are handled? What edge cases?

### Step 3: Detect Patterns

- Architecture patterns: MVC, hexagonal, clean arch, microservices
- Coding conventions: naming, error handling, logging, configuration
- Security patterns: auth middleware, input validation, rate limiting
- Performance patterns: caching, pagination, batch processing

### Step 4: Find Tests

- Read existing tests to understand expected behaviors and edge cases
- Map tests to source functions/modules they cover
- Identify untested areas (coverage gaps)

### Step 5: Generate Spec

Write the reverse-engineered spec to `.specify/specs/<branch>/spec.md`:

```markdown
# Spec: <branch-name> (Baseline)
<!-- Generated by /speckit.baseline from existing code -->
<!-- Date: YYYY-MM-DD -->
<!-- Source: <target module/directory> -->

## Overview
<inferred description of what this module/system does>

## User Scenarios

### US1: <Inferred scenario name> [P1]
- **Given** <inferred from code>
- **When** <inferred from code>
- **Then** <inferred from code>

## Functional Requirements

| ID | Requirement | Priority | Scenario | Source |
|----|-------------|----------|----------|--------|
| FR-001 | <requirement> | P1 | US1 | `path/to/source:42` |

## Success Criteria

| ID | Criterion | Validation Method | Existing Test |
|----|-----------|-------------------|---------------|
| SC-001 | <criterion> | <method> | `test/file.test.ts` or "None" |

## Coverage Gaps
- <areas with no tests>
- <behaviors without clear specifications>
- <error paths not covered>

## Detected Patterns
- Architecture: <pattern>
- Conventions: <notable patterns>
- Security: <protections in place>

## Clarifications
<!-- Ambiguities found during code analysis -->
- [NEEDS CLARIFICATION] <ambiguity found in code>
```

### Step 6: Present Summary

Report to the user:
- Files analyzed: N source files, N test files
- Scenarios extracted: N user scenarios
- Requirements inferred: N functional requirements
- Coverage gaps found: N areas without tests
- Clarifications needed: N ambiguities

### Suggested Next Steps

- **Review the generated spec** for accuracy — automated inference may miss business context
- **Run `/speckit.clarify`** to identify additional ambiguities
- **Run `/speckit.plan`** to design enhancements or refactoring
- **Run `/speckit.checklist`** to validate requirement quality
- **For new features on this code**: the baseline spec provides context for `/speckit.specify`
