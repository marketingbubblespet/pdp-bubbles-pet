# Spec-Kit: Brainstorm

Explore and refine the idea before writing a formal spec: **$ARGUMENTS**

## Pre-Flight

### Current branch
!`$HOME/.claude/hooks/speckit-helper.sh branch`

### Project context
!`$HOME/.claude/hooks/speckit-helper.sh detect-stack`

### Existing specs
!`$HOME/.claude/hooks/speckit-helper.sh list-specs-dir`

## Hard Gate

**NO CODE, SCAFFOLDING, OR IMPLEMENTATION UNTIL THE DESIGN IS APPROVED BY THE USER.**

This applies to ALL projects — "simple" ones included. Thoughtful design precedes execution, reducing wasted work from unexamined assumptions.

The ONLY exit from this command is handing off to `/speckit.specify`. No other implementation work is permitted.

## Instructions

### Step 1: Explore Context

- Understand the current project state: tech stack, existing patterns, conventions
- Read relevant existing code if the idea touches existing modules
- Identify constraints: what's already built, what can't change, what dependencies exist
- Note: this is orientation, not implementation — read-only

### Step 2: Ask Clarifying Questions (Socratic Dialogue)

Ask probing questions to refine the idea. **One question per message** during this phase.

Prefer **multiple-choice questions** over open-ended ones — they're faster to answer and expose trade-offs:

```
What's the primary goal here?
a) Speed up an existing workflow
b) Add a capability that doesn't exist yet
c) Replace something that's not working well
d) Something else — tell me more
```

**Question categories to cover** (not all will apply):

| Category | Example Questions |
|----------|-------------------|
| **Problem** | What problem does this solve? Who experiences it? How often? |
| **Users** | Who are the users? What do they know? What's their workflow today? |
| **Scope** | What's the minimum that would be useful? What's explicitly out of scope? |
| **Constraints** | Performance requirements? Security considerations? Backwards compatibility? |
| **Prior art** | Has this been attempted before? What worked/didn't? Are there similar systems? |
| **Success** | How will you know this works? What does "done" look like? |
| **Failure modes** | What happens when it fails? What's the worst case? |

**Stop asking when** you have enough clarity to propose concrete approaches — usually 3-5 questions. Don't over-interview; the spec phase captures remaining details.

### Step 3: Propose 2-3 Approaches

Present distinct alternatives with explicit trade-offs:

```
## Approach A: <name>
- How it works: <1-2 sentences>
- Pros: <specific advantages>
- Cons: <specific disadvantages>
- Best when: <conditions where this shines>
- Complexity: Low / Medium / High

## Approach B: <name>
- How it works: <1-2 sentences>
- Pros: <specific advantages>
- Cons: <specific disadvantages>
- Best when: <conditions where this shines>
- Complexity: Low / Medium / High

## Recommendation: Approach <X>
- Why: <specific reasoning tied to this project's constraints>
```

**Always recommend one.** Users can override, but a clear recommendation is more useful than "it depends."

### Step 4: Write Design Document

After the user selects or refines an approach, write a design document:

```markdown
# Design: <topic>
<!-- Generated by /speckit.brainstorm -->
<!-- Date: YYYY-MM-DD -->

## Problem Statement
<What problem this solves and why it matters>

## Chosen Approach
<The selected approach and why it was chosen over alternatives>

## Key Decisions
| Decision | Choice | Rationale |
|----------|--------|-----------|
| <decision> | <choice> | <why> |

## Scope
### In Scope
- <what's included>

### Out of Scope
- <what's explicitly excluded and why>

## Open Questions
- <any remaining unknowns — these become [NEEDS CLARIFICATION] in the spec>

## Alternatives Considered
| Alternative | Why Not |
|-------------|---------|
| <approach> | <reason it was rejected> |
```

**Design scales to complexity**: short for simple projects (50-100 words), detailed for nuanced ones (200-300 words). Match the actual scope.

### Step 5: Self-Review

Before presenting to the user, review the design inline:

- **Completeness**: Any TODOs, placeholders, or incomplete sections?
- **Consistency**: Any internal contradictions or conflicting requirements?
- **Clarity**: Any ambiguous points that could lead to misinterpretation?
- **Scope**: Appropriately focused for a single planning cycle?
- **YAGNI**: Does it avoid unrequested or over-engineered features?

**Apply a HIGH BAR** — only flag issues that would genuinely impair specification writing. Minor stylistic differences don't matter. **Approve unless significant gaps exist.**

### Step 6: User Review

Present the design document to the user. Wait for explicit approval before proceeding.

- If the user approves → proceed to handoff
- If the user has feedback → iterate on the design (return to relevant step)
- If the user wants to explore a different approach → return to Step 3

### Step 7: Handoff to Specification

After approval, suggest the next step:

> Design approved. Run `/speckit.specify <feature description>` to generate the formal specification with user scenarios, functional requirements, and success criteria.

The design document provides context for the specification phase — key decisions, scope boundaries, and chosen approach will inform the spec.

## Rationalization Prevention

| Thought | Why It Fails | Correct Action |
|---------|-------------|----------------|
| "This is too simple to brainstorm" | Simple projects fail from unexamined assumptions | At minimum: 2-3 clarifying questions |
| "I already know the right approach" | Confirmation bias skips better alternatives | Propose 2-3 approaches anyway |
| "The user just wants it done fast" | Fast wrong work is slower than slow right work | Ask first, implement once |
| "I'll figure it out as I code" | Leads to rework, scope creep, and dead ends | Design first, code second |
| "Let me just start with a prototype" | Prototypes become production code — always | Design the real thing from the start |

## Pipeline Position

```
/speckit.brainstorm → /speckit.specify → /speckit.plan → /speckit.review → /speckit.tasks → /speckit.implement
```

This command is the recommended starting point for any non-trivial feature. It ensures you're solving the right problem before defining requirements.
